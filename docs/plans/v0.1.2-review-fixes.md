In `src/output/format.ts`, replace `formatTruncationHint`:

```ts
/**
 * Human-mode hint emitted after a list when the fetch response hits the
 * requested limit. Returns `null` when no truncation signal is present.
 *
 * `rawCount` is the raw fetch count (pre any client-side filter).
 * `shownCount` is what was rendered to the user (defaults to rawCount).
 * When `shownCount < rawCount`, a client-side filter reduced the display
 * — the hint says so instead of "Showing first N items" (which would be
 * false in that case).
 *
 * `narrowingFlags` advertises flags that would actually change the fetch.
 * Omit client-side filter flags here — they don't narrow pagination.
 */
export function formatTruncationHint(
    rawCount: number,
    limit: number,
    narrowingFlags?: readonly string[],
    shownCount?: number,
): string | null {
    if (limit <= 0) return null;
    if (rawCount < limit) return null;
    const shown = shownCount ?? rawCount;
    const action = "Use --limit N to fetch more";
    const narrow =
        narrowingFlags && narrowingFlags.length > 0
            ? `, or narrow with ${narrowingFlags.join("/")}`
            : "";
    const lead =
        shown < rawCount
            ? `Filter matched ${shown} of ${rawCount} fetched.`
            : `Showing first ${limit} items.`;
    return `${lead} ${action}${narrow}.`;
}
```

## Task 6: Meta sequence flip (M1)

**Why:** Current sequence was `review → amend → tag`. Amend changes SHA, so tagged SHA != reviewed SHA. Flipping to `amend → review → tag` means the trailer-bearing SHA is what gets reviewed, and if clean becomes the tagged SHA.

Also rolls in the re-review-loop clarification I already wrote (uncommitted in meta/WORKFLOW.md).

**Files:**
- Modify: `/Users/dongzhenye/projects/meta/WORKFLOW.md`
- Modify: `/Users/dongzhenye/projects/meta/AGENTS.md`

- [ ] **Step 6.1: Rewrite sequence arrow in WORKFLOW.md**

In `meta/WORKFLOW.md` §3 Release Flow, replace the sequence block with:

````markdown
```
branch → implement → bump → self-review → fix
     → amend Co-Reviewed-By (on the release commit)
     → cross-model review
     → (if findings) fix → re-amend → re-review ...loop until clean
     → tag → push → publish → release
```

**Why amend precedes the final review pass**: if you amend after review, `git commit --amend` rewrites the SHA — the reviewed commit and the tagged commit no longer share an identity, and the `codex review --commit <SHA>` audit trail points at a SHA nobody ships. By amending the Co-Reviewed-By trailer FIRST and then running review on that SHA, a clean review result applies to exactly the SHA that will be tagged.

The `fix → re-amend → re-review → ...` loop is mandatory, not optional: any substantive fix produces a new commit whose trailer is stale (refers to a review against an earlier SHA). Re-amend and re-review until the HEAD SHA is what the reviewer actually approved.
````
